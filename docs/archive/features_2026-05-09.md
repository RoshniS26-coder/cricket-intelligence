## Future features (planned)

Distilled from the Engineering and Architecture notes; nothing in this
section is built yet.

### Analysis coverage
- **Bowler-side analysis** — currently the engine extracts bowler attributes per ball but the analytics pipeline only operates on batsmen. Mirror `weakness.py` and the pitch map for bowlers (over/wicket density, line-and-length effectiveness vs particular batsmen).
- **Wicketkeeper role model** — the schema can already represent keeper
  events; an analytics module + UI tab need to follow.
- **Per-shot coaching context** in critiques — today the multi-shot session
  applies the same `--coaching-keys` set to every shot section. Phase 2:
  pick the right coaching extracts per shot type automatically.
- **Continuous video-watch mode** — a tail-style ingestion that consumes a
  growing video file (or a stream) instead of completed MP4s.

### Pose / CV
- **Pose threshold calibration** — `calibrate_pose_thresholds.py` is
  referenced in `ENGINEERING.md` but not yet implemented; would derive
  per-camera thresholds from a labeled set instead of hard-coded
  `head_offset < 0.03`.
- **Synced impact-frame side-by-side renderer**
  (`render_comparison_video.py`) — aligns the impact frame across student
  and reference, then stacks with pose-skeleton overlay.
- **Custom YOLO ball + stumps detector** — replace the hosted Roboflow
  inference with a fine-tuned local model (the training pipeline at
  `CV_Enhancements/yolo_training/` is ready; just needs a labeled set).

### Continuous learning
- **Ground-truth flywheel UI** — the `ground_truth` table exists
  (`migrate_add_pose.py`) but no UI yet writes coach corrections back into
  it. Streamlit form + API endpoint to capture each review.
- **Re-prompting from corrections** — periodically sample the
  `ground_truth` table to A/B-test prompt updates.

### Data hygiene (asked for, not yet executed in this pass)
- **Stale match cleanup** — DB rows for `test_match_001..007`,
  `demo-match`, `NZvsSA-1-match` were preserved in this restructure pass
  but are obvious cleanup targets when the team is ready.
- **Old-video cleanup** — `data/raw_videos_old/` (35 files) and the
  multi-GB videos in `data/raw_videos/` likewise.
- **Extracted-JSON cleanup** — `data/*_extracted.json` exports duplicate
  what's in the DB.

## Recently-applied Gemini-prompt improvements

Landed in this restructure pass — listed for traceability, not as roadmap:

1. **Granular shot rubric in single-ball prompt** — previously only the
   batch prompt had the drive/cut/sweep family rubric, so single-ball
   extractions defaulted to broad `drive` / `cut`. Rubric now inlined into
   `EXTRACTION_PROMPT`.
2. **Left-hander flip rule** for `swing_direction` and `spin_direction` in
   both single-ball and batch prompts — labels emitted in the batter's
   reference frame.
3. **Name-confidence fields** — `confidence.bowler_name` and
   `confidence.batsman_name` added to schema. Lets the weakness analyzer
   drop low-confidence joins instead of poisoning danger zones with bad
   OCR.
4. **`runs_scored` bounded** to `[0, 6]` with explicit description (no
   more byes/leg-byes counted as boundaries).
5. **CV partial-detection guard** in `extractor.py` — line/length CV
   override now requires both ball and stumps detected with confidence
   ≥ 0.6.
6. **Replay-timing heuristic** in `BATCH_EXTRACTION_PROMPT` — gives Gemini
   the "5–25s after a live ball is almost certainly a replay" rule of
   thumb to apply when scoreboard OCR is unreliable.
